RFCAudit Skill 设计方案

把 RFCAudit 仓库中调用 LLM 的 Python 脚本，转化为一个 opencode 原生 skill：codegraph 替代 tree-sitter，opencode 自身替代 OpenAI API 调用，扁平子代理编排替代 autogen 多智能体。

Approach A · codegraph-native 项目级 .opencode/skills/ 无嵌套子代理 5 路并行

0脚本 → Skill 替换映射

opencode 本身就是 LLM agent。原脚本中对外的 API 调用与 autogen 编排，全部由 opencode 原生能力承担。

原脚本做的事替换为

query_repo_recursive.pytree-sitter 解析 + query_name/callercodegraph_explore（含调用路径/动态分发）

APhase A — RFC 切分与摘要

输入用户提供 RFC 文档原文；输出章节全文归档 + 章节摘要索引。

切分规则

1
按章节号正则切割匹配 ^(\d+(\.\d+)*)\s+(标题)，原子粒度为 2 级（如 2.1、2.2、3.2）。
2
更深层并入父节2.1.1 的内容归入 2.1；仅有顶层 2（无子节）也保留为独立单元。
3
LLM 生成摘要对每个 2 级章节输出一段总结，概括该章节描述的行为与约束。
降级方案（fallback）：若文档不含章节号，则按 Markdown 标题层级切分——以 ##（h2）为原子单元（等价于 2 级）。若连标题也无，整篇作为单一单元处理。

产物

全文与 JSON 分离：JSON 只存 title / summary / content_path。

// RFC/{protocol}/rfc_sections.json { "RFC 2460": { "3": { "title": "IPv6 Header Format", "summary": "定义40字节固定头，含版本/流标签/负载长度...", "content_path": "RFC/ipv6/sections/RFC2460_3.md" }, "4": { /* ... */ } } }

BPhase B — 代码切分、摘要与匹配

把代码目录树切分为可摘要的单元，对每个单元总结"做什么"并匹配到 RFC 章节。这一步既产出代码摘要，又完成范围裁剪。

目录切分规则

1
根目录 = 第 1 级默认切分到 第 3 级目录为原子单元。
2
>100 源码文件则下沉拆到第 4 级；仍 >100 拆到第 5 级（上限）。
3
确定性预扫描纯文件计数，无 LLM；主 agent 先产出完整目录清单，再均分派发。
下沉示例：某 3 级目录含 300 源码文件 → 下沉到 4 级子目录 → 某子目录仍 120 文件 → 继续到 5 级（上限）。

均分算法：预扫描得到有序目录列表 dirs[0..N-1]，并行度 k（默认 5）。第 i 个分片取 dirs[i*N/k .. (i+1)*N/k)，按个数均分而非按文件数——因为成本驱动是"待摘要的目录数量"。分片边界对齐目录树的连续区间，保持同子树的目录尽量在同一分片（共享上下文）。

每个子代理对分配的每个目录执行：① 读头文件/关键源码（codegraph_explore 采样）② 写目录摘要 ③ 对照 RFC 章节摘要判断关联度 ④ 打置信度分。

CPhase C — 审计（核心）

对 code_map 中每个 high/medium 关联的（目录 × RFC 章节）执行深度代码审查，找出实现与规范的显式不一致。主 agent 扮演原 autogen GroupChatManager，扁平派发分析与批判子代理。

编排流程（扁平，无嵌套）

批量同步并行（bulk-synchronous）：每批内——先并行派发 batch_size 个分析子代理（一条消息多个 task），等全部返回；再并行派发对应数量的 critic 子代理，等全部返回；合并确认结果写入 JSON，进入下一批。分析与批判分两轮，因 critic 需读取对应分析结果，二者不可在同一轮并行。
为什么按目录而非按（目录×章节）对：同一目录的多个关联章节共享同一次 codegraph 代码探索，按目录聚合可避免重复探索。若某目录过大（关联章节极多），可降级为按章节拆分该目录的分析任务。

分析子代理的职责

输入：RFC 章节全文 + 目录路径。用 codegraph 递归探索代码后判定不一致。

1
理解规范从章节提取强制行为、约束、要求；只考虑显式声明的，不推断。
2
限定范围探索代码用 codegraph_explore 仅在该目录范围内取相关函数/宏/类型定义；取调用者上下文作佐证。探索范围由 Phase B 映射限定——不发散到全项目。
3
严格比对只报告对强制行为的显式违反。检查调用点是否已保证前置条件。
不报告：可选/未定义行为、合理的实现选择、日志 vs 静默处理差异。本阶段只识别问题，不出修复建议。
范围限缩原则：Phase B 已为每个目录标注关联的 RFC 章节。审计时 codegraph 探索以该目录为边界，只取目录内的符号；调用路径跟随若跨出目录（如调用方在另一目录），仅取该调用点作上下文佐证，不展开成对越界目录的全面审计。每个目录的越界关联若确实相关，应由 Phase B 的匹配结果体现——审计不自行扩大范围。

critic 子代理的职责

输入：分析结果 + 原始 RFC 章节 + 代码。验证分析的正确性、完整性、有效性。

1
验证探索充分性确认所有相关代码路径都经 codegraph 探索，含调用点逻辑与约束。
2
校验每条不一致确认是对规范强制行为的明确违反；排除误报。
3
最终裁决有效→确认；无效→附理由驳回；存疑→建议进一步调查路径。
排除误报来源：可选/未定义行为、可接受的实现策略、日志差异、规范中不存在却被推断出的要求、调用方已保证的前置条件。

codegraph 在审计中的角色

原 tree-sitter 操作codegraph_explore 替代额外能力
query_function(name)按函数名取定义源码同时给出调用关系
query_caller(name)blast radius / 调用路径含动态分发跳点
query_type / query_def按 struct/宏名取定义类型引用图
init(project) 全量解析索引已存在，即查即用无预扫描开销
调用路径能力 vs 审计范围：codegraph 能力上可跨任意目录追踪调用路径（相对 tree-sitter 的优势）；但审计策略上，探索以 Phase B 映射的目录为边界——跨目录调用仅取调用点作上下文佐证，不展开成全面审计。

!关键设计约束与决策

扁平编排（无嵌套）

实验验证：子代理工具集无 task。并行只在一层发生（主 agent 同时派 N 个独立子代理）；analyze↔critic 是顺序两步派发。

可选：通过 experimental.primary_tools + permission.task 可实验性开启子代理的 task 工具，但有递归爆炸风险。

前置条件

目标项目须有 .codegraph/ 索引。skill 首步检查；若无则指示用户运行 codegraph init。

config.yaml 不再需要

协议名、项目路径、RFC 文件路径由用户调用时以参数给出，或在对话中询问。LLM 调用无需 API Key（用 opencode 自身模型）。

裁剪策略

采用 LLM 摘要匹配（Phase B）替代纯关键字/图可达性裁剪——更可解释，摘要可复用。关键字漏掉的共享基础设施（如 rtalloc1）在 Phase C 审计时由 codegraph 按调用路径补全。
